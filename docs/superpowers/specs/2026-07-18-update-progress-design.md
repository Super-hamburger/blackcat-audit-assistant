# 更新流程进度反馈设计

## 目标

让已安装软件在检查和安装更新时始终给出清晰、中文且不中断界面的反馈。用户能够看到当前阶段、下载进度、失败原因，并能在失败后重试。

## 范围

- 手动检查更新改为异步执行，避免网络请求阻塞主窗口。
- 一键更新的下载、SHA256 校验和解压改为异步执行。
- 以自定义中文对话框替代默认 `QMessageBox` 的英文 `Yes/No` 确认框。
- 新增阶段式进度窗口，显示检查、下载、校验、解压和准备重启状态。
- 更新失败时保留进度窗口，显示可理解的原因并提供重试和关闭操作。
- 保持后台自动检查为静默模式：只更新设置页状态，发现新版本时沿用托盘通知。

不在本次范围内：自动发布 GitHub Release、改变更新包格式、修改远程更新清单协议，或在源码运行模式下覆盖项目文件。

## 用户流程

1. 用户点击“检查更新”。按钮立即禁用，状态显示“正在检查更新…”。
2. 检查完成后恢复按钮，并更新当前版本、最新版本和状态文字。
3. 若有可安全安装的新版本，显示中文更新摘要对话框，提供“现在更新”和“稍后”。
4. 用户选择“现在更新”后，进度窗口显示“正在下载更新包”，并显示百分比、已下载大小和总大小（服务器提供长度时）。
5. 下载完成后，依次显示“正在校验更新包”“正在解压更新包”“正在准备安装”。
6. 准备成功后，显示“更新已准备完成”，用户确认后程序退出，由现有独立安装脚本覆盖文件并重新启动。
7. 任一步骤失败时，窗口显示失败阶段和错误原因；用户可选择“重试”或“关闭”。

## 架构与职责

- `core/update_manager.py`：继续负责读取清单和比较版本；不接触界面。
- `core/update_installer.py`：增加可注入的进度回调，在下载、校验和解压时报告结构化阶段事件；保留现有安全校验和更新脚本职责。
- `core/update_notification.py`：增加后台工作对象，向主线程发送检查结果、更新阶段和最终结果。
- `ui/widgets/`：新增可复用的中文更新确认窗和进度对话框；对话框负责文本、进度条、重试/关闭按钮和可访问名称。
- `ui/main_window.py`：只负责启动后台任务、禁用/恢复操作控件、渲染状态并在确认后启动现有更新脚本。

## 状态模型

检查状态：`idle`、`checking`、`update_available`、`up_to_date`、`check_failed`。

安装状态：`downloading`、`verifying`、`extracting`、`ready_to_restart`、`failed`。

每个安装事件包含用户可见中文说明；下载事件另外包含 `downloaded_bytes`、`total_bytes` 和可选百分比。未知总大小时使用不确定进度条，仍显示已下载大小。

## 失败处理

- 网络、HTTP、清单解析失败：说明检查失败，不把本地清单当作“已是最新”。
- 下载失败：保留下载 URL 与错误文本，允许重试完整下载。
- SHA256 不匹配：立即停止，不启动安装脚本，并明确提示校验失败。
- ZIP 解压或安装包结构异常：立即停止并提示安装包无效。
- 源码运行：保留“一键更新只在打包安装后的软件中执行”的提示，不下载或覆盖项目文件。

## 可用性与无障碍

- 所有用户可见按钮使用中文：`现在更新`、`稍后`、`重试`、`关闭`、`确认并重启`。
- 使用应用现有主题的明确前景色，避免深色弹窗上出现不可读的深色正文。
- 进度窗口显示文字阶段，不只依赖颜色或动画。
- 更新期间禁止重复发起同一操作；失败和关闭后恢复操作控件。

## 验证

- 单元测试：版本检查成功/失败、可更新/已最新状态、下载进度事件、未知内容长度、SHA256 失败和解压失败。
- 界面测试：手动检查不阻塞窗口；所有确认按钮为中文；失败后可重试；成功后只在用户确认时退出。
- 发布前按项目约定运行静态检查、JSON 校验、相关测试和打包 EXE 的 `--self-test`。
